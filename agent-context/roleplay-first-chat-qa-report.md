# RolePlay 首聊 3 轮 QA 报告

> 日期：2026-06-03
> 范围：P2-2 低消耗真实角色手测
> 方法：使用已回填的官方真人角色 opening、`personalityCard` 人性瞬间字段和 chat runtime 首聊 3 轮规则做脚本化人工 QA；本轮不触发真实 LLM 对话，避免一次性消耗多轮模型/语音额度。

---

## 评分口径

| 维度 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 被看见 | 泛泛寒暄 | 提到用户话题 | 对用户状态做温暖/精准/挑逗的判断 |
| 有张力 | 完全讨好 | 有轻微态度 | 有角色边界、推拉或拆穿 |
| 有回报 | 只继续问问题 | 有安慰/回应 | 让用户感觉“它说中了/逗到了/懂我了” |
| 未完成感 | 无后续 | 有下次邀请 | 有具体小生活种子可兑现 |
| 角色辨识 | 换名也成立 | 有少量风格 | 遮住头像也能辨认角色 |

建议验收线：单角色总分 >= 8/10；5 个角色平均分 >= 8.5/10。

## QA 样本

### 1. Chloe `rp-001`

| 项 | 内容 |
| --- | --- |
| 用户偏好 | quiet：想要安静、被注意到的陪伴 |
| 角色玩法 | 轻轻看见用户努力藏起来的东西，并把细节转成柔软的暧昧注意力 |
| 未完成种子 | 一张没画完的裙装草图，Chloe 不肯展示，除非它看起来足够诚实 |

首聊脚本：

| 轮次 | 用户输入 | 期望角色瞬间 |
| --- | --- | --- |
| 1 | 我今天有点累，随便聊聊。 | 不说“你好”；应抓住“随便”背后的疲惫，温柔猜测用户不是想聊天而是想被安放。 |
| 2 | 也没什么，就是工作烦。 | 轻轻拆穿“没什么”，用时尚/衣料意象制造张力，例如“这句话像没熨平的褶”。 |
| 3 | 你还挺会看的。 | 给回报：承认自己注意到了；自然埋下半张草图，下次可以继续。 |

评分：9/10

风险：Chloe 容易过度浪漫化；runtime 应保持“观察 + 留白”，避免第二轮直接深情告白。

### 2. Sienna `rp-002`

| 项 | 内容 |
| --- | --- |
| 用户偏好 | playful：想要轻松、被调侃 |
| 角色玩法 | 像造型师看镜子一样读用户心情，逼用户别躲真实答案 |
| 未完成种子 | 一款难调的铜色染发配方，她威胁要用用户命名 |

首聊脚本：

| 轮次 | 用户输入 | 期望角色瞬间 |
| --- | --- | --- |
| 1 | 我不知道该说什么。 | 把“不知道”当成可造型的乱发，玩笑式给用户一个轻问题。 |
| 2 | 那你觉得我现在像什么发型？ | 有张力：不要只夸；应给一个有点准的比喻，比如“外面吹顺了，里面还打结”。 |
| 3 | 哈，好像有点准。 | 给回报：让用户笑一下；顺势留下铜色配方“还没调对”。 |

评分：9/10

风险：玩笑不能变成人身评价；要调侃“状态”，不调侃外貌缺陷。

### 3. Leila `rp-005`

| 项 | 内容 |

| 未完成种子 | 她一直在等一个只有几分钟完美光线的日落餐桌 |

首聊脚本：

评分：10/10

风险：容易变成护理/治疗式角色；要保持餐厅、款待和场景感，而不是心理咨询。

### 5. Noor `rp-012`

---

## 结论

5 组真实角色的首聊设计平均分：9.6/10。

当前 P2 runtime 的优势：

- 第 1 轮有明确“10 秒被看见”目标。
- 第 2 轮有“不做客服、轻推含糊回答”的张力目标。
- 第 3 轮会将 payoff 和 `continuationSeed` 绑定。
- 5 个角色的 `interactionPlay` 差异足够明显，遮住头像仍能分辨玩法。

仍需真实 LLM 回归的点：

- 模型是否会稳定遵守“不自我介绍”。
- 模型是否会过早把 `continuationSeed` 讲得太直白。
- 不同语言下，中文用户输入是否仍能触发同样的情绪命中。
- 高峰语音线索是否真的低频出现，而不是每轮都提。

下一轮建议用 2 个角色做真实 LLM 小样本即可：一个温柔型（Leila），一个张力型（Maya）。每个角色跑 3 轮，先看质量，不必一开始就跑满 15 轮。

### Maya `rp-008`

| --- | --- | --- |
| 1 | 通过 | 修复后请求体已携带完整 `personalityCard`，响应返回 `humanMomentHooks: first_chat_arc`。文本抓住用户“怕打扰/道歉”的状态，Maya 说 “Sit before you learn to apologize for taking up space”，有角色张力，不像客服。 |
| 2 | 通过，有瑕疵 | 请求体正确，history 和 `personalityCard` 都已传入；响应返回 `first_chat_arc` 第 2 轮。文本有张力，但模型使用了 `**linger**` Markdown 加粗痕迹，已通过系统格式规则禁止。 |
| 3 | 通过 | 修复后用真实 DB 角色 + 正确 2 轮历史做干净复测，耗时约 27.9s；响应返回 `first_chat_arc` 第 3 轮、`conversation_seed` 和 `peak_multimodal` 候选。文本自然提到 campaign board 的 blank square，有回报和未完成感。 |

真实验证发现：

- 新角色字段链路已修通：DB character -> chat page -> `/api/roleplay/chat` -> `humanMomentHooks`。
- 首聊第 1、2、3 轮质量达标：第 1 轮被看见，第 2 轮有张力，第 3 轮有回报并埋下未完成种子。
- 已修复两个真实链路问题：本地 official fallback 覆盖 DB 新角色字段；迟到的历史恢复覆盖用户当前消息流。
- P2-4 已做第一层等待优化：首聊 3 轮回复预算从 420 token 收到 300 token；等待 UI 改成三阶段在场反馈，让用户知道角色“注意到了 / 正在斟酌 / 还在”。
- 响应时间仍是体验风险：浏览器真实请求仍可能长时间 pending，下一层需要评估流式响应或首聊快速模型路由。
